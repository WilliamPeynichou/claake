# V0.04 — Architecture d'exécution des agents (2026-03-16)

## Résumé

Claake passe d'un modèle **clé IA globale côté serveur** à un modèle **marketplace** : chaque agent définit sa propre stratégie d'exécution, choisie par le vendeur. Le backend Claake ne détient plus jamais de clé IA. Support de 14 fournisseurs IA officiels.

## F2 — Service de chiffrement (cross-cutting)

### Port

- `backend/src/common/ports/encryption.port.ts`
- Token : `ENCRYPTION_SERVICE = Symbol("ENCRYPTION_SERVICE")`
- Interface : `encrypt(plaintext) → string`, `decrypt(encrypted) → string`

### Implémentation

- `backend/src/common/services/aes-encryption.service.ts`
- **AES-256-GCM** via `node:crypto`
- IV aléatoire (12 bytes) par appel
- Format stockage : `base64(iv):base64(authTag):base64(ciphertext)`
- Clé depuis `ENCRYPTION_KEY` env var (32 bytes hex = 64 chars)

### Wiring

- Enregistré `@Global()` dans `AppModule` via `providers` + `exports`
- Nouvelle env var : `ENCRYPTION_KEY` ajoutée au `.env`

## F4 — Refonte providers IA (cœur du changement)

### Principe

Suppression de toute clé IA globale côté serveur. Chaque requête IA reçoit sa clé en paramètre, déterminée par la stratégie de l'agent.

### StreamTextParams

- Ajout `apiKey?: string` (clé par requête, déjà déchiffrée)
- Ajout `baseUrl?: string` (pour SELLER_ENDPOINT et providers custom)

### AnthropicProvider

- Suppression `ConfigService` et clé globale
- Utilise `params.apiKey` dans le header `x-api-key`
- Erreur si pas de clé fournie

### OpenAIProvider

- Suppression `ConfigService` et clé globale
- Utilise `params.apiKey` dans `Authorization: Bearer`
- Support `params.baseUrl` pour providers OpenAI-compatible

### EndpointProxyProvider (nouveau)

- `backend/src/modules/chat/infrastructure/providers/endpoint-proxy.provider.ts`
- Proxy vers l'endpoint du vendeur
- **3 familles de parsing** :
  - **OpenAI-compatible** (SSE standard) — openai, mistral, deepseek, groq, xai, perplexity, meta, together, fireworks, huggingface, claake
  - **Anthropic** (SSE `content_block_delta`) — anthropic
  - **Google Gemini** (JSON `generateContent`) — google
  - **Cohere** (SSE `text-generation` / `content-delta`) — cohere
- Protection SSRF : timeout 60s, `redirect: "error"`, taille max 10MB

### ExecutionStrategyResolver (nouveau)

- `backend/src/modules/chat/application/services/execution-strategy.resolver.ts`
- Token : `EXECUTION_STRATEGY_RESOLVER = Symbol(...)`
- Logique centrale :

| Situation | Action |
|-----------|--------|
| Agent LOCAL-only | Erreur "Téléchargez l'agent" |
| Agent sans `cloudStrategy` | Erreur "Agent non configuré" |
| `SELLER_ENDPOINT` | → EndpointProxyProvider + endpointUrl/format |
| `SELLER_API_KEY` | → Déchiffre clé vendeur → AnthropicProvider ou OpenAIProvider |
| `USER_API_KEY` | → Déchiffre clé user → AnthropicProvider ou OpenAIProvider |

### SendMessageUseCase

- Remplace `aiFactory.getProvider(model)` par `strategyResolver.resolve(agent, userId)`
- Le resolver retourne `{ provider, extraParams }` injectés dans `streamText()`

### ChatModule

- Ajout : EndpointProxyProvider, ExecutionStrategyResolver, ManageApiKeysUseCase
- Import : UserModule (pour accéder aux clés user)
- Conserve : AnthropicProvider, OpenAIProvider, AIProviderFactory (utilisés par le resolver)

## F5 — Gestion clés API utilisateur (CRUD backend)

### UserRepositoryPort

- Ajout `getApiKeysEncrypted(userId)` et `setApiKeysEncrypted(userId, data)`
- Implémenté dans PrismaUserRepository (stockage JSON dans `users.api_keys_encrypted`)

### ManageApiKeysUseCase

- `backend/src/modules/users/application/usecases/manage-api-keys.usecase.ts`
- Implémente `ManageApiKeysPort` (interface du module chat)
- Méthodes :
  - `listKeys(userId)` → liste masquée (`sk-a1...b2c3`)
  - `addKey(userId, provider, label, key)` → chiffre AES-256-GCM + sauvegarde
  - `removeKey(userId, keyId)` → supprime par ID
  - `getDecryptedKeyForProvider(userId, provider)` → utilisé par ExecutionStrategyResolver
- Format stockage JSON : `[{ id, provider, label, encryptedKey, createdAt }]`

### Endpoints (AuthController)

| Méthode | Route | Auth | Description |
|---------|-------|------|-------------|
| GET | `/v1/auth/api-keys` | Oui | Liste masquée des clés |
| POST | `/v1/auth/api-keys` | Oui | Ajouter une clé (body: `{ provider, label, key }`) |
| DELETE | `/v1/auth/api-keys/:id` | Oui | Supprimer une clé |

## Modes d'exécution

| Mode | Description |
|------|-------------|
| **CLOUD** | Exécution côté serveur uniquement |
| **LOCAL** | 100% client-side, l'acheteur télécharge Docker/config |
| **HYBRID** | Supporte CLOUD + LOCAL |

### Stratégies cloud (vendeur choisit)

| Stratégie | Description |
|-----------|-------------|
| **SELLER_ENDPOINT** | Le vendeur expose son API, Claake proxy vers elle |
| **SELLER_API_KEY** | Le vendeur fournit sa clé IA, stockée chiffrée en BDD |
| **USER_API_KEY** | L'acheteur fournit sa propre clé dans ses settings |
