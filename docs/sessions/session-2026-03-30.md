## 1. Fix bug envoi de messages

**Symptôme** : impossible d'envoyer un message — Entrée ne répondait pas, clic sur Send non plus.

**Cause racine** : le layout `(dashboard)` applique `padding: 1.5rem` sur `<main>`. La page chat utilisait `h-[calc(100vh-4rem)]` sur son conteneur, ce qui créait un dépassement de 24px en bas. Le composant `ChatInput` était rendu hors de l'écran, invisible et non cliquable.

**Fix** — `frontendWeb/app/(dashboard)/chat/layout.tsx` :
```tsx
// Marges négatives pour compenser le padding du parent
return <div className="-mx-6 -my-6 flex h-[calc(100vh-4rem)] overflow-hidden">{children}</div>;
```

## 3. Pipeline multimodal — Upload image/PDF + analyse IA

### 3.1 Schéma Prisma — nouveau modèle `UploadedFile`

```prisma
enum UploadedFileType { IMAGE  DOCUMENT }

model UploadedFile {
  id        String           @id @default(uuid())
  url       String           // URL publique Supabase Storage
  mimeType  String
  type      UploadedFileType
  size      Int              // en octets
  userId    String
  agentId   String?          // fichier lié à un agent
  sessionId String?          // fichier lié à une session chat
  messageId String?          // fichier lié à un message
  createdAt DateTime         @default(now())
}
```

Appliqué via `prisma db push` (`migrate dev` échouait — shadow DB incompatible avec la migration `0001_init`).

### 3.2 Module Uploads (NestJS)

Nouveau module `backend/src/modules/uploads/` :

**`UploadService`** :
- Valide le type MIME : `image/jpeg`, `image/png`, `image/webp`, `image/gif`, `application/pdf`
- Limite : 10 Mo max
- Upload vers Supabase Storage (bucket `agent-files`)
- Crée l'enregistrement `UploadedFile` en DB

**`UploadController`** — endpoints :
```
POST   /v1/uploads?agentId=&sessionId=   → upload multipart → { id, url, type, mimeType }
GET    /v1/uploads/agent/:agentId        → liste fichiers d'un agent
GET    /v1/uploads/session/:sessionId    → liste fichiers d'une session
DELETE /v1/uploads/:fileId              → suppression (vérification propriétaire, 204)
```

### 3.3 Bucket Supabase Storage

Bucket `agent-files` créé via Supabase REST API (le MCP n'expose pas d'endpoint direct).

**Fix critique** : la `SUPABASE_SERVICE_ROLE_KEY` dans `backend/.env` avait un caractère corrompu dans le payload JWT (`ultbfmrxeey` → `ultbfmrxyey`). Toutes les requêtes Storage retournaient 403 "signature verification failed" jusqu'à la correction.

### 3.4 Support multimodal dans le provider Anthropic

**`ai-provider.port.ts`** — nouvelle interface :
```typescript
export interface FileAttachment {
  type: "image" | "document";
  url: string;
  mimeType: string;
}
```

**`anthropic.provider.ts`** — construction des blocs Anthropic :
```typescript
// Blocs prépendés au dernier message utilisateur
{ type: "image", source: { type: "url", url: "..." }, ... }
{ type: "document", source: { type: "url", url: "..." }, ... }
```

**`send-message.usecase.ts`** — paramètre `attachments: FileAttachment[]` ajouté.

**`chat.controller.ts`** — résolution `file_ids` → `FileAttachment[]` via `prisma.uploadedFile.findMany`.

### 3.5 Format SSE backend (Vercel data stream)

Le contrôleur NestJS émet le protocole Vercel AI data stream :
```
0:"texte du chunk"\n         → token de texte
d:{"finishReason":"stop"}\n  → fin de stream
3:"message d'erreur"\n       → erreur
```
Headers : `Content-Type: text/plain; charset=utf-8`, `x-vercel-ai-data-stream: v1`

## 5. Fix compatibilité `@ai-sdk/react` v3

### Problème

`@ai-sdk/react` v3.0.143 a supprimé les props `input`, `handleInputChange`, `handleSubmit` de `useChat`. L'appel `input.trim()` levait une `TypeError` car `input` était `undefined`.

### Cause

L'API v3 expose `sendMessage({ text })` + `status` à la place de l'ancien `handleSubmit` + `isLoading`. Les types `Message` sont aussi remplacés par `UIMessage` avec `parts[]` au lieu de `content: string`.

### Solution retenue

Retour au hook `useChat` de `@claake/shared/hooks` qui :
- Gère le streaming SSE manuellement (compatible avec le format backend)
- Expose `input`, `setInput`, `streaming`, `sendMessage` — API stable
- Supporte `pendingFileIds`, `addPendingFile`, `removePendingFile`

Les composants UI (`ChatMessage`, `ChatThread`, `ChatInputDA`) utilisent maintenant `ChatMessage` de `@claake/shared` au lieu des types AI SDK.

## Architecture du pipeline multimodal

```
Utilisateur sélectionne un fichier
        ↓
ChatInputDA → POST /v1/uploads?sessionId=xxx
        ↓
UploadController → UploadService
        ↓
Supabase Storage (bucket: agent-files) → URL publique
        ↓
UploadedFile enregistrée en DB (avec sessionId)
        ↓
Utilisateur envoie le message avec file_ids=[uuid1, ...]
        ↓
ChatController → prisma.uploadedFile.findMany({id: [uuid1]})
        ↓
FileAttachment[] → SendMessageUseCase → AnthropicProvider
        ↓
Blocs Anthropic: [{ type:"image", source:{type:"url", url} }, { type:"text", text }]
        ↓
Réponse streamée → SSE (0: d: 3:) → useChat → UI
```
